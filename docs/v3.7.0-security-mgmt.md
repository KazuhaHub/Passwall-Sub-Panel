# v3.7.0 账号安全管理增强（设计稿）

> 本稿为 v3.7.0 收官阶段「账号安全管理」批次的设计真相源。功能发布后删除本稿（代码是真相源）。

## 背景

v3.7.0 已实现 验证码/锁定/邮箱找回/自助注册/2FA/Passkey（beta.1~8）。实际使用中暴露出一批管理与体验缺口，本批次集中收口。

## 需求清单（用户 2026-06-06 提出）

1. 管理员能集中管理某用户的凭证：2FA / Passkey / 备用码 / 重置密码……当前散在「更多」菜单（已 8 项，过载）。
2. 登录 2FA 挑战不要把备用码和 TOTP 混在一个输入框；做「使用其他方式验证」——可选 **备用码 / Passkey / Email OTP**，**管理员可勾选启用哪几种**。用户也能重新生成备用码。
3. 默认邮件模板 logo 裂开。
4. 验证器 App 里显示的 issuer 应是网站名（`Kazuha Hub Passwall`），而非 `Passwall`。
5. 验证码不应只用于登录，按场景勾选：登录 / 注册 / 找回密码。
6. 邮件模板编辑界面缺「重置密码」「创建账户(验证邮箱)」两个模板。
7. 模板变多后横向子 tab 不合理，需换布局。

## 关键现状（探查结论）

- **品牌名两个字段**：`SiteTitle`（侧栏/页头品牌名，默认 `Kazuha Hub Passwall`）vs `AppTitle`（产品名，默认 `Passwall`）。TOTP issuer（[twofa.go:103-106](../internal/service/twofa/twofa.go)）、Passkey RP 显示名（[passkey.go:95](../internal/service/passkey/passkey.go)）、邮件头品牌（[mailer.go:823/880/1262](../internal/service/mailer/mailer.go)）都误用 `AppTitle`。
- **邮件 logo 非 bug**：模板引用正确，资源 `/images/logo+title-circle-darkmode.png` 存在且 `/images/` 公开 serve。裂开是因 `SubBaseURL=http://127.0.0.1:8788`，Gmail 图片代理够不到 localhost。文件名里的 `+` 对部分邮件代理不友好。
- **备用码无重新生成**：仅 Enable 时一次性给，丢了只能管理员重置整个 2FA + 用户重新绑定。
- **Passkey 不能当 2FA 替代因子**：开了 TOTP 后即使 passkey 登录仍强制 TOTP；`/auth/2fa/verify` 只认 TOTP/备用码。
- **验证码仅登录**：[auth_local.go:289](../internal/transport/http/handler/auth_local.go) 唯一校验点；register/forgot 无。仅全局开关 `CaptchaEnabled`+`CaptchaTrigger`。
- **邮件模板**：后端 `DefaultTemplates()` 已含 `password_reset`/`email_verify`（[mailer.go:284-295](../internal/service/mailer/mailer.go)），`ListTemplates`/重置默认/校验都覆盖；前端 tab 列表写死 8 个漏了这俩（[settings.ts:277](../web-react/src/api/settings.ts)、[SettingsView.tsx:1024-1136](../web-react/src/views/admin/SettingsView.tsx)）。

## 设计决策（已与用户对齐）

### A. 品牌名统一
人脸可见的品牌名一律：`SiteTitle` 优先 → 回退 `AppTitle` → 回退 `"Passwall"`。抽一个 helper `brandName(settings) string`（放 ports 或各自包内小函数），覆盖 TOTP issuer、Passkey RP 显示名、邮件头。不动 App 头部既有的 `AppTitle` 用法。

### B. 邮件 logo 硬化
- 资源重命名 `logo+title-circle*.png` → `logo-title-circle*.png`（URL/代理安全）。同步改 render.go、mailer.go、index.html、public/images 与 dist。
- 文档 + 设置提示：邮件 logo 要外部可见需配公网 `SubBaseURL` 或绝对 `LogoURL`；localhost 永远裂（Gmail 不代理本机）。

### C. 2FA 备用码重新生成
- `twofa.Service.RegenerateRecovery(ctx, userID, code) ([]string, error)`：自助，需现 TOTP/备用码证明持有；生成新 10 码、替换旧 hash、明文返回一次。
- `twofa.Service.AdminRegenerateRecovery(ctx, userID) ([]string, error)`：管理员破窗，无需证明（与重置密码同级），明文返回一次供转交。
- 端点：`POST /user/me/2fa/recovery/regenerate`（RequireAuth+loginLimiter）；`POST /admin/users/:id/2fa/recovery/regenerate`（staffGroup+ensureOperatorAllowed）。
- 前端：自助在 TwoFactorDialog 增「重新生成备用码」步；管理员在「账号安全」抽屉里。

### D. 2FA 替代验证方法矩阵
登录第一因子（密码或 passkey）通过后，若 `TOTPEnabled` 返回 `{status:"2fa_required", pending_token, methods:[...]}`。

| method | 启用条件 |
|---|---|
| `totp` | 基线（始终） |
| `recovery` | 始终（兜底，不做开关） |
| `passkey` | `twofa_allow_passkey` 开 + passkey 功能开 + 用户有 passkey + **第一因子==password** |
| `email` | `twofa_allow_email` 开 + 用户有邮箱 |

- **pending token 增 `ff`(first_factor) claim**：`password`|`passkey`。passwordless passkey 登录时不提供 passkey-2FA（避免同因子两次）。
- **可用 methods 服务端算**：新增 `twofa`/handler 协作算出 methods 列表放进 `2fa_required` 响应；前端据此渲染「使用其他方式验证」。
- **Passkey-2FA**：`POST /auth/2fa/passkey/begin` `{pending_token}` → 解析 pending、校验 ff==password & 开关 & 有 passkey、按用户 allow-list 发起断言（新增 `passkey.BeginLoginForUser(userID)`/`FinishLoginForUser`）；`POST /auth/2fa/passkey/finish` 验断言→完成登录（复用 loginResponse）。共用 loginLimiter。
- **Email-2FA**：`POST /auth/2fa/email/send` `{pending_token}` → 校验 ff、开关、有邮箱 → 发 OTP 邮件（新 `auth_tokens` purpose `login_2fa`，短 TTL ~10min，5 次烧毁，限流），返回通用 200；验证走扩展后的 `/auth/2fa/verify`：TOTP → recovery → **login_2fa OTP**（按 userID 查 auth_token，匹配则消费并完成）。
- **管理员开关**：设置「登录与安全 → 两步验证」加 `twofa_allow_passkey`(默认 true)、`twofa_allow_email`(默认 false)。recovery 始终可用不做开关。
- **新增 mail kind**：`login_2fa`（2FA 邮箱验证码）默认模板，纳入可编辑。

### E. 验证码多场景
- 设置拆 per-context：`captcha_login_enabled`+`captcha_login_trigger`、`captcha_register_enabled`、`captcha_forgot_enabled`（注册/找回用 always 语义即可，不做 after-failures；登录保留 trigger）。
- 向后兼容：旧 `CaptchaEnabled`/`CaptchaTrigger` 若新字段未设，迁移为 login 场景（保持现状）。
- 校验点：register（[auth_register.go](../internal/transport/http/handler/auth_register.go)）、forgot（[auth_recovery.go](../internal/transport/http/handler/auth_recovery.go)）加 captcha.Verify；forgot 失败仍返回 200（防枚举）。
- `/auth/methods` 拆 `captcha_login_required`/`captcha_register_required`/`captcha_forgot_required`。
- 前端 RegisterView/ForgotPasswordView 各自挂 CaptchaWidget，按对应字段 gate。
- 失败计数沿用跨场景池（不加 auth_event 列），仅阈值/开关 per-context。

### F. 管理员「账号安全」抽屉
- 右侧抽屉（MUI Drawer），标题=用户 UPN。分区：登录密码（重置）/ 两步验证（状态+重置+重新生成备用码+剩余数）/ 通行密钥（列表+单个/全部吊销）/ SSO 绑定（解绑）/ 订阅凭证（重置凭证 UUID）/ 紧急访问（重置计数）。
- 「更多」菜单瘦身为：复制订阅 URL / 个人规则 / 账号安全…（其余安全动作迁入抽屉）。
- 管理员**不能代为新增** 2FA/passkey（无用户设备）；只能禁用/重置/吊销/重新生成备用码。

### G. 邮件模板编辑器
- 前端补 `password_reset`/`email_verify` 到类型、列表、i18n、变量提示（增 `{{.ResetLink}}`/`{{.VerifyLink}}`/`{{.OTPCode}}`）。
- 布局横向 tab → master-detail：左侧分组列表（运营提醒 / 账号安全）+ 右侧编辑器。
