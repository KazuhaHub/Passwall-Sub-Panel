## 决策日志（v2 相对 v1 的变化）

| 议题 | 决策 | 依据 |

| 整体方法 | **保持**「PSP 自持 ACME + DNS-01 + 内联 PEM 下发」 | prior-art 对抗 verdict=`keep-with-tweaks`，`betterMethodExists=false`：3X-UI 无节点侧签证书 API；同类面板全靠节点 agent（我们没有）；这是 API-only 约束下的天花板 |

| 域名不覆盖 | **统一只警告**（含 psp_managed），允许保存 | 用户选定 |
| 证书策略 | 同时支持「每节点独立」与「一张通配符全节点共用」，**UI 默认推荐通配符** | 用户认可默认 |

## 0. 背景与硬约束

PSP 只能通过**别人的** 3X-UI 的 HTTP API 操作节点，**无节点文件系统/SSH/agent**，API 也无"写文件"端点。由此：

| 做法 | 可行 | 原因 |
|---|---|---|
| cert 文件落节点磁盘按路径引用 | ❌ | 够不着节点盘 |
| HTTP-01 验证 | ❌ | 域名解析到节点，PSP 不在那台机 |
| 委派节点自己 acme.sh 签发 | ❌ | 3X-UI 的 acme 只在宿主机 shell（`x-ui cert` CLI / 菜单），`/panel/api/*` 无任何签发端点（源码核对 server.go/api.go），HTTP-01 only 无 wildcard |
| **DNS-01 签发 + 内联 PEM 下发** | ✅ | 与节点位置无关、支持 wildcard；Xray 原生接受内联 `certificate`/`key` PEM 数组；PSP 控制 inbound 配置 |

**同类对照**：Marzban / marznode / Hiddify / s-ui 全靠节点 agent（有 FS 权限）写文件 + 热加载；唯一对得上的是 **Remnawave**（面板持证书、下发时把内容塞进配置推给节点）—— 我们就是这个模型，只不过走 3X-UI inbound API。**重启代价是 API-only 的固有成本，不是缺陷。**

## 1. 三种「证书来源」（节点表单逐节点选择器）

| `cert_source` | 行为 | 前置 | 生命周期 |
|---|---|---|---|
| `manual`（含空=历史默认） | 手填 file 路径 / inline PEM | 无 | PSP 不管 |
| `from_panel` | 调 3X-UI `getWebCertFiles` 拉**路径**回填 file 模式 | 3X-UI **3.2.7+**，旧版按钮置灰+提示 | PSP 不管（节点 acme.sh 自行续，PSP 只是省手敲；享 Xray 每小时文件热加载，无重启） |
| `psp_managed` | 逐节点绑一张 PSP 托管证书（`cert_id`），存盘/续期把 PEM **内联**下发进 inbound | 配 DNS 凭据 + 签发成功 | **PSP 自动续期 + 内容 diff 后重下发** |

> `getWebCertFiles` 是 3.2.7 才有的端点（源码逐条核对 v3.2.6/3.2.7）。`checkAPIAuth` 接受 Bearer，PSP 现有 per-node token 即可调，无需新增鉴权。它**只返回路径、永不返回 PEM 字节**，故只能服务 `from_panel`，不能替代 `psp_managed` 的内联。

## 2. ACME 库：`go-acme/lego/v4`

- MIT，~150 家 DNS provider 内置 DNS-01。`certificate.Obtain(ObtainRequest{Domains, Bundle:true})` 返回 `*certificate.Resource`，其 `Certificate []byte`（Bundle 时 leaf+issuer）、`PrivateKey []byte` **已是 PEM**，直接内联进 `streamSettings.tlsSettings.certificates[]`。
- **不用 certmagic**：它为"本进程 listener 用证书"优化（tls.Config.GetCertificate + 内存 cache + 集群锁），要用得自写 Storage 把 PEM 抠出来，纯属逆库。
- **lego 完全封在 `internal/adapters/acme/`**，domain/service/ports 不 import lego。
- **DNS provider（精选 + 兜底，避免全家桶）**：**不** import lego 的 `providers/dns` aggregator（它会把全部云 SDK 拉进依赖树、二进制胖几十 MB）。改用**精选 ~20 家显式注册表**（cloudflare/alidns/dnspod/route53/gcloud/azuredns/godaddy/namecheap/namesilo/porkbun/digitalocean/gandiv5/hetzner/linode/ovh/vultr/desec/duckdns）+ **exec/httpreq 通用兜底**（长尾走脚本/webhook，零云 SDK）。每厂商一行 `code → NewDNSProvider`（env-based）；凭据 map 注入对应 env 后 `Obtain` 全程 **mutex 串行**消除 env 竞争。`acme.SupportedProviders()` 暴露给前端下拉（S4/S7）。实现于 [internal/adapters/acme/lego.go](../internal/adapters/acme/lego.go)。
- go.mod 新增 `github.com/go-acme/lego/v4`（精选 provider 的传递依赖含 aws/gcp/azure/alibaba SDK；`alidns` 是其中最重的，不用可删一行 import + registry）。

  ID PK | Name | Domains jsonStrings(含 wildcard) | ACMEAccountID idx | DNSCredentialID idx

## 5. 业务服务 `internal/service/cert/`

`Service` 组合：`CertificateRepo` / `DNSCredentialRepo` / `ACMEAccountRepo` / `ACMEIssuer` / `NodeRepo` / `SyncTaskRepo` / 节点下发链路 / `mailer`。

- **CRUD**：DNS 凭据、证书。`CreateCert` 落 `pending` 行后 `enqueueCertTask(SyncTaskCertIssue, certID, ...)`。
- **任务处理 `ProcessDueTasks(ctx, limit)`**（仿 node）：`ListDue`→`MarkRunning`→`runCertTask`→成功 `MarkSucceeded` / 瞬时错 `MarkRetry(+backoff)` / 永久错 `Cancel`+置 `status=failed`。
  - **错误分类**：缺 DNS 凭据/账户配置错=**永久**（Cancel，status=failed，告警）；DNS 没传播/LE 限额/网络=**瞬时**（MarkRetry）。
- **`runCertTask`**：`Obtain` → 持久化（`UpdateIssued`：cert/key/notBefore/notAfter/fingerprint，status=active；账户 key 回填则 `ACMEAccountRepo.Save`）→ 对每个绑定节点 `DeployToNode`。
- **`DeployToNode(ctx, node, cert)`**：**内容 diff 门控（T2）**——比节点现存证书 `Fingerprint`，相同则跳过；不同才把 PEM 注入 `StreamSettings.tlsSettings.certificates[0].certificate/.key` 并走**现有下发链路**。
  - 复用确认：[NodesView.tsx](../web-react/src/views/admin/NodesView.tsx) `buildTLSSettings` 已产出内联 PEM 数组；[reconcile.go](../internal/service/reconcile/reconcile.go) 已有**保客户端**的 read-modify-write `UpdateInbound`。`DeployToNode` 复用，不新造推送。
  - **域名校验（决策=仅警告）**：cert 域名不覆盖节点 `ServerAddress`/`tls_server_name` 时记 warning，**不阻断**。
- **`ScanDueRenewals(ctx)`**（worker 调）：遍历 `AutoRenew` 证书，按**混合阈值**判定到期，`enqueueCertTask(SyncTaskCertRenew, ...)`。
  - **混合阈值**：寿命 `L = NotAfter-NotBefore`，剩余 `R = NotAfter-now`，配置 `N` 天。
    `T = (N > (2/3)L) ? L/3 : N`；当 `R ≤ T` 触发续期。（短效证书 N 过大→自动落到剩 1/3 兜底，避免抖动空续。）

## 11. 关键约束与权衡

1. **内联换证必重启 Xray（秒级断流）**：Xray 对内联强制 `oneTimeLoading=true`，只有 file 路径模式才有每小时热加载（XTLS PR#281）。这是 API-only 固有成本。缓解（T4）：① 同节点多 inbound 合并一次 `UpdateInbound`/重启；② 低峰续期；③ 跨节点错峰；④ 内容 diff 门控（T2）不空放；⑤ 每节点显示「剩 N 天/上次下发状态」。
2. **续期阈值用混合公式**（§5），存 `NotBefore`/`NotAfter`，别写死天数。
3. **LE 限额**：开发期先接 staging directory；账户按 (email,directory) 复用；证书持久化重启不重签；瞬时错退避。续期不计入每周限额。
4. **ACME Obtain 串行**（mutex）；常用厂商显式 config 免 env 竞争。
5. **私钥必须下发到节点**（节点终止 TLS）：cert+key 全 `encryptSecret` 落盘，且 `StreamSettings` 走 PSP 既有加密传输。
6. **3X-UI bug #4651**：UI/client 编辑会清空 inbound 的 `certificateFile`/`keyFile`。`psp_managed` 内联自带在 PSP 配置里、每次 push 全量重置 streamSettings，天然规避；`from_panel`（file 路径）由节点续期、PSP 不常 push，定期重申留作开放项。
7. **自用无迁移**：新表/列走 AutoMigrate。
